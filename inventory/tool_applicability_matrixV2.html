Runner grouping

Groups by tenant → site. Multiple instances per client (multiple WSUS, FortiGates, ADs, vCenters) are each a separate playbook entry under their site. Results are merged at the client level before writing — a device seen in two sites writes the most recent value.

vCenter — virtual appliances

pfsense, FortiMail and similar VMs have device_role=appliance in Netbox even though they live in vCenter. The runner matches them as VMs (virtualization/vms endpoint) by vm_name (guest OS hostname custom field), then display name, then IP.

vCenter — match priority

For VMs: vm_name (Netbox custom field = guest hostname) first, then name (vCenter display name), then validate all returned IPs against Netbox primary_ip. ESXi hosts match by IP (host field in script).

FortiGate — MAC fallback

Hostname is not always populated in FortiGate ARP/DHCP tables. IP is primary match. MAC address is last resort — matched against Netbox interface MACs via dcim/interfaces or virtualization/interfaces.

Elastic — appliances via syslog

All appliances must send logs to Elastic. cf_elastic_last_seen for appliances is updated from syslog ingestion events, not an installed agent. elastic_agent_ver will be null for these devices.

Nessus — workstations

Currently excluded from Nessus scan policy. Create a separate scan profile or adjust subnet scope to include workstation network. Business rules will handle per-client exclusions — benchmark keeps all types in scope.

GLPI scope

GLPI is AD-synced — only covers server and workstation devices. Appliances and VMs not present unless manually added. No open_tickets field — removed from scope.

Client layer override

This matrix is benchmark defaults. Per-client exclusions go in playbook.yaml under rules:. Business overrides never modify this file — it stays stable as the benchmark reference.
